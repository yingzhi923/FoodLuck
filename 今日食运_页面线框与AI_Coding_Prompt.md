- 地图主界面
- 学生友好
- 性价比导向

# 四、页面线框说明

## 4.1 首页（主地图页）

### 页面目标
用户进入产品后，第一眼理解：

- 这是一个校园周边美食地图
- 核心功能是“今日食运”
- 也可以通过“许愿”筛选餐厅
- 可以直接看地图和推荐卡片

### 页面布局建议

#### 顶部导航栏（固定）
从左到右：

- **学校 / 大学城切换入口**
  - 示例文案：松江大学城｜东华大学
  - 点击后弹出切换面板
- **中间主按钮：今日食运**
  - 视觉上最突出
  - 建议做成圆角胶囊按钮或带轻微拟物感的签筒按钮
- **搜索按钮**
  - 圆形 icon
- **头像按钮**
  - 圆形头像，点击进入个人页（可做占位）

#### 地图主区域（页面上半部分 55%~65%）
地图为首页核心视觉区，功能包括：

- 展示餐厅点位 marker
- 当前用户位置（可模拟固定点）
- 点击 marker 高亮相应卡片
- 卡片点击时地图中心移动到对应餐厅

#### 底部结果面板（页面下半部分，可拖拽）
默认状态下展示附近餐厅卡片。

建议面板顶部设置两个模式切换 tab：

- **附近**
- **许愿**

“今日食运”不放在 tab 中，而作为顶部最强入口。

### 首页默认展示内容

- 顶部：品牌 + 入口
- 中部：地图
- 底部：附近推荐卡片流

### 首页推荐卡片字段
每张卡片建议包含：

- 店名
- 菜系标签
- 人均价格
- 步行时间
- 学生标签（1-2 个）
- 一句推荐理由
- 按钮：查看详情 / 加入签筒

### 首页交互重点

1. 点击地图 marker → 底部卡片自动切到对应店
2. 滑动底部卡片 → 地图高亮对应店
3. 点击“今日食运” → 打开食运弹层
4. 点击“许愿”tab → 进入许愿筛选模式
5. 点击卡片 → 进入商家详情页

## 4.2 今日食运弹层 / 页面

### 页面目标
帮助“不知道吃什么”的用户在最短时间内获得一个可接受的结果。

### 设计原则

- 快速开始
- 可控随机
- 有情绪价值
- 结果可执行

### 建议流程

#### Step 1：快速条件选择
采用按钮 chips 形式，尽量避免复杂表单。

条件建议：

- **步行范围**：5 分钟 / 10 分钟 / 15 分钟
- **预算**：20 以内 / 30 以内 / 50 以内 / 80 以内
- **类型**：正餐 / 小吃 / 甜品 / 夜宵
- **人数**：1 人 / 2 人 / 3-4 人
- **忌口/不想吃**：不吃辣 / 不吃面 / 不想吃汉堡 / 不要连锁

页面底部按钮：

- 生成候选签

#### Step 2：候选签筒生成
系统根据条件展示 4-6 个候选项。

候选项展示方式建议：

- 横向签条 / 标签球 / 列表 chip
- 每个候选项带删除 icon
- 页面底部支持新增自定义项

示例：

- 螺蛳粉
- 兰州牛肉面
- 卤肉饭
- 韩式拌饭
- 日式咖喱饭

用户可以：

- 删除不想要的项
- 手动添加一个想纳入随机池的选项

页面底部按钮：

- 开始摇签

#### Step 3：摇签动画
建议轻量实现，不需要复杂 3D。

可落地形式：

- 签条滚动
- 圆盘轻微旋转
- 模糊候选项快速切换 2-3 秒

动画期间可出现文案：

- 今日食运加载中...
- 你今天的口福正在生成...

#### Step 4：结果页
结果页建议包含：

- 今日运势：大吉 / 小吉 / 平签
- 抽签结果：餐厅名或菜品名
- 推荐理由：如“步行 6 分钟，人均 28，适合一个人快速解决晚饭”
- 餐厅卡片摘要
- 按钮：
  - 换一签
  - 一键导航
  - 查看详情

### 食运结果的产品细节建议

增加两个小机制提升体验：

#### 1. 最近吃过过滤
避免连续抽到相似餐厅。

#### 2. 不喜欢此结果
用户点“不喜欢”后，系统减少类似店铺出现概率（Prototype 中可只做交互占位）。

## 4.3 许愿页面 / 模式

### 页面目标
服务“有大致方向，但不知道具体去哪家”的用户。

### 页面布局建议
页面上半部分保留地图，下半部分展示结果卡片。

#### 筛选区域（置顶）
使用 chips + 下拉/弹层。

推荐筛选项：

- 用餐人数：1 人 / 2 人 / 3-4 人 / 聚餐
- 人均预算：20 内 / 30 内 / 50 内 / 80 内 / 100+
- 目标距离：步行 5 分钟 / 10 分钟 / 15 分钟
- 菜系：中餐 / 日料 / 韩餐 / 泰餐 / 西式 / 甜品 / 咖啡 / 夜宵
- 场景：一个人吃 / 朋友聚餐 / 约饭不踩雷 / 赶时间 / 聊天坐坐 / 夜宵

#### 补充输入框（可选）
不是主输入方式，只是补充说明。

占位文案示例：

- 想吃点热的
- 想安静一点
- 想找适合聊天的地方

#### 推荐结果区
根据条件给出 3-5 家餐厅卡片。

每张卡片建议显示：

- 店名
- 菜系
- 人均
- 步行时间
- 学生标签
- 推荐理由
- 按钮：查看详情 / 去这里

### 许愿模式交互重点

1. 修改筛选条件后即时刷新结果
2. 点击卡片时地图同步定位
3. 点击“去这里”调用导航
4. 地图与卡片形成联动

## 4.4 商家详情页

### 页面目标
让用户在做决定前快速获得足够的信任信息。

### 页面模块

#### 头图区域
- 店铺封面图
- 店名
- 菜系
- 人均价格
- 距离 / 步行时间

#### 学生友好标签区
建议展示 3-5 个标签，例如：

- 20 元吃饱
- 一个人也能吃
- 约饭不踩雷
- 夜宵救命
- 适合赶时间

#### 店铺简介区
简单介绍该店适合什么场景。

#### 推荐菜 / 推荐理由区
可以列 2-3 个推荐菜名，并给一句学生视角推荐理由。

#### 操作区
按钮建议：

- 一键导航
- 加入签筒
- 收藏（可选）

### 详情页信息原则
不要堆大量信息，要优先展示影响决策的内容：

- 贵不贵
- 远不远
- 适不适合当前场景
- 值不值得去

# 五、核心组件设计建议

## 5.1 顶部导航栏
应具备：

- 固定吸顶
- 高辨识度品牌按钮
- 操作简洁

## 5.2 餐厅卡片组件
建议统一组件化，供首页、许愿页、搜索页复用。

字段统一：

- 店名
- 人均
- 步行时间
- 菜系标签
- 学生标签
- 推荐理由
- CTA 按钮

## 5.3 Filter Chips 组件
所有条件筛选尽量用 chips，而不是复杂表单。

原因：

- 更适合移动端和轻量网页
- 更像消费决策产品
- 操作门槛更低

## 5.4 食运签条组件
这是品牌识别组件，建议重点设计。

要点：

- 看起来有“抽签”感
- 支持删除
- 支持新增
- 能配合动画

## 5.5 底部抽屉面板
首页推荐使用 Bottom Sheet 形式承载结果。

优点：

- 与地图产品习惯一致
- 方便地图和卡片联动
- 能切换不同模式内容

# 六、视觉与体验建议

## 6.1 风格关键词

- 年轻
- 轻松
- 友好
- 明快
- 不要过于商务
- 不要太像大众点评复刻版

## 6.2 品牌调性建议
“今日食运”要有一点点东方签运感，但不要做得太玄学。

建议：

- 用签筒做品牌 icon
- 运势文案轻松俏皮
- 保持克制，不要过度传统国风

## 6.3 页面情绪化文案示例

首页：
- 今天想吃什么？不如交给食运。
- 30 秒，决定你的下一顿。

食运中：
- 今日食运正在生成...
- 你今天的口福有点旺。

结果页：
- 大吉：今天适合吃点热乎的。
- 小吉：这顿稳，不容易踩雷。
- 平签：普通的一天，也值得认真吃饭。

许愿页：
- 说说你想吃什么，我来帮你找。

## Prompt 开始

请帮我开发一个前端网页 Prototype，产品名叫 **“今日食运”**，这是一个服务大学生的“大学城美食地图”产品。

### 一、产品定位
这是一个帮助大学生快速决定“今天吃什么”的地图型网页应用。核心目标不是做一个完整点评平台，而是做一个 **低后悔、低门槛的校园周边吃饭决策工具**。

用户痛点：
1. 大学校园周边餐饮很多，但信息分散、选择困难
2. 学生用户更关注性价比、步行距离、是否适合当前场景，而不是泛化评分
3. 吃遍周边后容易陷入“今天不知道吃什么”的决策疲劳

### 二、核心功能
网页需要重点实现两个核心场景：

#### 1）今日食运
用于“不知道吃什么”的用户。
这是产品最核心的功能，带一点“电子抽签”的趣味感。

功能流程：
- 用户点击顶部主按钮“今日食运”
- 弹出一个食运面板
- 用户通过 chips 选择快速条件：
  - 步行范围：5 分钟 / 10 分钟 / 15 分钟
  - 预算：20 内 / 30 内 / 50 内 / 80 内
  - 类型：正餐 / 小吃 / 甜品 / 夜宵
  - 人数：1 人 / 2 人 / 3-4 人
  - 忌口/不想吃：不吃辣 / 不吃面 / 不想吃汉堡 / 不要连锁
- 点击“生成候选签”后，系统展示 4-6 个候选项
- 用户可以删除某些候选项，也可以手动新增一个候选项
- 点击“开始摇签”后，展示一个轻量动画（例如签条滚动切换）
- 最终显示抽签结果页：
  - 今日运势（大吉 / 小吉 / 平签）
  - 结果餐厅
  - 推荐理由
  - 两个按钮：换一签 / 一键导航
  - 还有查看详情入口

#### 2）许愿
用于“有偏好，但不知道去哪家”的用户。

功能流程：
- 用户在首页底部结果面板中点击“许愿”模式
- 页面展示筛选条件 chips：
  - 用餐人数
  - 人均预算
  - 步行距离
  - 菜系
  - 场景（一个人吃 / 朋友聚餐 / 约饭不踩雷 / 赶时间 / 聊天坐坐 / 夜宵）
- 提供一个可选的补充说明输入框，例如“想吃点热的”“想找安静一点的地方”
- 根据条件筛选，推荐 3-5 家餐厅
- 页面上半部分是地图，下半部分是推荐商家卡片
- 点击卡片时地图同步定位到对应 marker
- 卡片上有“查看详情”和“去这里”按钮

### 三、页面结构
请用现代 Web App 风格来设计，整体偏移动端 / 窄屏网页风格，但也适配桌面端。

#### 1）首页（主地图页）
页面包含：
- 顶部固定导航栏
  - 左侧：大学城 / 学校切换入口，例如“松江大学城｜东华大学”
  - 中间：最突出主按钮“今日食运”
  - 右侧：搜索按钮、头像按钮
- 中间：地图区域（需要预留高德地图容器 div）
- 底部：可拖拽/固定式结果面板（bottom sheet）
  - 默认显示“附近推荐”餐厅卡片
  - 可切换到“许愿”模式

#### 2）商家详情页
包含：
- 店铺封面图
- 店名、菜系、人均、步行时间
- 学生标签（例如“20 元吃饱”“一个人也能吃”“约饭不踩雷”）
- 推荐理由
- 推荐菜
- 操作按钮：一键导航、加入签筒

#### 3）搜索页（可简单）
包含：
- 搜索框
- 热门搜索
- 搜索结果列表

#### 4）个人页（可弱化）
只做简单静态页面即可，包含：
- 当前学校
- 收藏的店
- 最近抽签记录
- 饮食偏好

### 四、视觉风格要求
- 年轻、轻松、清爽
