# 今日食运｜迭代需求说明 v3

## 一、文档信息

- **项目名称**：今日食运｜大学城美食地图
- **版本类型**：迭代需求说明 / PRD 变更版
- **适用阶段**：当前原型已跑通后的功能与体验升级
- **目标**：在不推翻现有代码架构的前提下，完成品牌强化、主入口重构、会员日功能补充、抽签与许愿体验优化

---

## 二、当前产品基线（基于现有代码架构）

当前项目已经具备以下基础能力：

1. **启动与进入方式**
   - 项目基于 `Vite + React + React Router` 构建。
   - 应用启动后先进入 `Onboarding`，用户完成学校选择后进入首页。

2. **首页结构**
   - 首页由 `Index.jsx` 承载。
   - 页面结构为：顶部导航栏 `TopNav` + 地图 `MapView` + 底部热榜 `BottomSheet`。
   - 首页地图目前仅展示首页精选点位，底部为“附近热榜”展示。

3. **今日食运主入口**
   - 顶部按钮会打开 `FortuneModal`。
   - 当前 `FortuneModal` 已具备模式选择与子流程结构：
     - 模式页（目前已有“抽一签”“许个愿”）
     - 抽签流程（选条件 → 看候选 → 摇签 → 结果）
     - 许愿流程（复用 `WishPanel`）

4. **数据基础**
   - 当前项目餐厅主数据统一来自 `src/data/restaurants.js`。
   - 全量 mock 餐厅数据已扩展至 200 家。
   - 筛选条件、标签体系、学校列表、预算/步行/人数等常量由 `src/data/constants.js` 提供。

5. **当前待优化问题**
   - 品牌识别尚未完全统一（landing page、顶栏、modal 内部图标存在不一致或未替换）。
   - 今日食运入口虽然已形成模式切换，但尚未加入“今日宜食”。
   - 抽签结果页缺少可点击的动作反馈。
   - 抽签返回逻辑不完整。
   - 许愿筛选项顺序尚未按用户真实决策路径优化。

---

## 三、本轮迭代目标

本轮迭代有三大目标：

### 目标 1：强化品牌感
让“今日食运”名称与 logo 在 onboarding / 顶栏 / modal / 抽签相关界面中形成统一识别。

### 目标 2：把“今日食运”做成真正的主决策入口
将“今日宜食”“抽一签”“许个愿”统一收口到一个清晰的三入口模式页中。

### 目标 3：提升决策闭环与可用性
完成会员日信息展示、抽签结果可点击提示、200 家商家接入筛选池、返回逻辑修复、许愿筛选顺序优化。

## 五、用户价值与产品定位更新

经过当前版本验证后，产品核心心智更新为：

> **今日食运 = 帮大学生更快决定今天吃什么的入口**

对应三种决策模式：

1. **今日宜食**：我想省钱，想知道今天哪里最划算。
2. **抽一签**：我懒得想，让系统替我决定。
3. **许个愿**：我已经有方向，希望按条件筛出更合适的选择。

# 6.3 新增功能：今日宜食

## 6.3.1 功能定义

### 核心定位
“今日宜食”是一个**按星期展示的会员日/优惠情报板**，不接入真实外卖/会员系统，不与当前 200 家餐厅数据直接打通。

### 目标价值
抓住大学生“追求性价比”的核心痛点，帮助用户用更低成本做出选择。

---

## 6.3.2 数据来源与处理方式

### 当前输入来源
用户已提供一份会员日文字清单（按周一到周日整理）。

### 产品要求
- 先做成**本地静态配置**，不连接接口。
- 不直接渲染原始 rtf 文本。
- 先将内容手动整理为结构化数据，例如：

```js
[
  {
    weekday: 1,
    title: '周一',
    items: [
      { brand: '麦当劳', offer: '会员日', ctaLabel: '一键前往' },
      { brand: '喜茶', offer: '免配送费', ctaLabel: '一键前往' }
    ]
  }
]
```

### 数据清洗要求
对原始文档中的重复项、异写项、格式异常项做基础清洗，例如：
- 合并重复品牌
- 去掉明显重复行
- 保留更适合展示的简洁优惠语
- 对存在星期记录歧义的内容，优先按人工整理后的结果配置

### 推荐新文件
- `src/data/memberDayOffers.js`
- 或 `src/data/todayDeals.js`

---

## 6.3.3 展示逻辑

### 展示规则
- 系统自动识别今天是周几。
- 用户点击“今日宜食”后，直接进入当天的优惠信息展板页。
- 不需要让用户再做额外筛选。

### 页面内容结构
1. 标题：今日宜食
2. 副标题：今天是周 X，这些品牌更划算
3. 内容区：当天优惠品牌列表
4. 底部说明：优惠信息仅供参考，以平台实时活动为准

### 单条卡片/按钮展示结构
每一行一个可点击按钮，包含：
- 品牌名
- 优惠信息
- CTA 文案：一键前往

### 点击反馈
本轮先不接跳转，只要具备点击交互即可。
可接受两种方式：
- 点击后无实际跳转，仅展示 hover / active 状态
- 或点击后弹出统一提示（推荐和抽签 CTA 统一体验）

---

## 6.3.4 交互要求

### 必须具备
- 自动按星期渲染内容
- 每一条优惠都是可点击按钮
- 内容可滚动，适配“周三信息更多”的情况

### 不需要实现
- 真正跳去外卖平台
- 与餐厅详情页或地图联动
- 收藏、分享、订阅提醒

### 验收标准
- 周一到周日每天都能显示对应内容。
- 页面不会因为某一天品牌太多而溢出布局。
- 用户明确感知“这是今天的优惠情报页”。

# 6.4 抽签功能优化

## 6.4.1 抽签结果页增加可点击 CTA

### 当前问题
抽签结果页只能看结果，缺乏“下一步动作”的反馈。

### 目标
让用户在结果页中感知到“可以一键前往”，即使当前还未真正接通导航。

### 方案要求
在结果页中新增明确 CTA：
- 文案建议：**一键前往** / **去看看**

### 点击后反馈
点击后弹窗或 toast 提示：

> 现在为您一键导航…（ps 此功能稍后上线哦）

### 产品说明
- 不建议让整个结果卡都变成点击区域。
- 建议将 CTA 做成单独按钮，语义更清晰。

### 验收标准
- 抽签结果页存在清晰 CTA。
- 用户点击后能收到占位提示反馈。
- 不会误触发重新摇签。

---

## 6.4.2 抽签池扩充到全量 200 家

### 当前问题
抽签与许愿仍可能只使用原始 45 家或旧池逻辑，未完全体现 200 家 mock 数据的价值。

### 目标
抽签候选池必须接入当前全量 200 家餐厅数据。

### 要求
- `FortuneModal` / `filterRestaurants` / `getRandomItems` 等逻辑应统一从全量 `restaurants` 数据源取值。
- 去重与历史规避逻辑仍然保留。
- 不改变当前抽签流程结构。

### 验收标准
- 抽签结果不再明显重复于旧的 45 家。
- 多轮抽签能覆盖更丰富的品类与店型。

---

## 6.4.3 抽签返回逻辑修复

### 当前问题
抽签流程中的返回按钮不可用，或返回行为不正确。

### 目标
抽签页返回后，应返回到“今日食运”模式页（即“今日宜食 / 抽一签 / 许个愿”三个按钮所在页面）。

### 层级要求
`FortuneModal` 需要建立清晰状态层级：
- 一级：模式选择页
- 二级：今日宜食 / 抽签 / 许愿
- 三级：抽签内部步骤页

### 返回规则
- 抽签步骤页的返回：回到模式选择页
- modal 右上角关闭：关闭整个 modal

### 验收标准
- 返回按钮可用。
- 抽签返回后不丢失 modal 上下文。
- 不会错误关闭整个弹窗。

## 6.5.2 许愿筛选项顺序调整

### 当前问题
筛选顺序与用户真实思考路径不一致。

### 新顺序要求
将“人数”和“场景”移动到前两项。

### 推荐顺序
1. 人数
2. 场景
3. 预算
4. 距离
5. 菜系
6. 补充说明（可选）

### 产品原因
大学生在约饭时，通常先想清楚：
- 几个人吃？
- 这顿饭是什么场景？
再进一步考虑预算、距离和菜系。

### 验收标准
- `WishFilters` 的筛选项顺序调整完成。
- 表单结构更符合用户决策逻辑。
