# 「今日食运」大学城美食地图 PRD

- **文档版本**：V1.0
- **产品阶段**：MVP / Prototype
- **文档类型**：前端展示型网页产品 PRD
- **适用对象**：产品设计、AI Coding、前端原型开发
- **作者角色预设**：C 端产品经理

---

## 1. 项目背景

某高校聚集地（如大学城、学区商圈）周边餐饮商户丰富，但学生群体在日常就餐时普遍面临以下问题：

1. **信息分散**：餐厅信息散落在外卖平台、点评平台、社交平台与线下口碑中，缺少一个围绕校园生活场景聚合的信息入口。
2. **决策疲劳**：大学生就餐频率高、熟悉周边商户后容易“吃遍一圈”，在选择时反而更容易陷入“不知道吃什么”的状态。
3. **缺少学生视角**：现有平台大多以大众消费场景为主，缺少针对学生预算、步行距离、宿舍生活节奏、拼饭需求、夜宵需求的推荐逻辑。
4. **高频场景未被优化**：学生用户更关注“便宜、近、快、适合当前情境”，而不是仅仅浏览海量餐厅列表。

因此，需要设计一款面向大学城学生群体的网页应用，通过地图化展示与轻决策机制，帮助用户在更短时间内决定“今天吃什么”。

---

## 2. 产品定位

### 2.1 产品名称

**今日食运**

### 2.2 一句话定位

一款服务大学生的校园周边吃饭决策地图，用预算、步行距离和偏好，帮用户更快决定今天吃什么。

### 2.3 核心价值主张

- **比大众平台更懂学生**：围绕大学生消费能力、时间成本和就餐情境进行推荐。
- **比纯地图更容易做决定**：地图提供空间感，卡片和推荐结果负责决策闭环。
- **比普通推荐更有记忆点**：通过“今日食运”有选择的随机机制，降低决策疲劳，提升产品趣味性。

---

## 3. 产品目标

### 3.1 MVP 目标

本产品首版不追求做成完整美食平台，而是验证以下核心能力：

1. 用户能在 **10 秒内理解产品用途**。
2. 用户能在 **30 秒内获得一个可执行的吃饭建议**。
3. 用户能从推荐结果中 **直接导航前往目标餐厅**。

### 3.2 核心体验目标

- 帮用户解决“今天吃什么”的决策问题。
- 提供围绕校园生活的本地化、学生化餐厅推荐。
- 建立“地图 + 轻决策 + 情绪价值”的差异化产品体验。

## 4. 目标用户

### 4.1 主要用户

大学城周边的在校大学生，包括：

- 日常独自吃饭的学生
- 与同学、室友拼饭聚餐的学生
- 预算敏感型用户
- 夜宵、赶课、宿舍生活场景用户
- 对周边餐厅已较熟悉但经常陷入选择困难的用户

### 4.2 用户画像

#### 用户画像 A：高频外卖 / 堂食决策困难型
- 平时吃饭频率高
- 对周边餐厅比较熟悉
- 常出现“都吃过了，不知道选什么”的困扰
- 需要低成本、低思考负担的决策工具

#### 用户画像 B：预算敏感型学生
- 关注人均价格与性价比
- 更在意“20 元能否吃饱”“值不值得专门走过去”
- 对平台的大众评分不敏感，更在意学生口碑和场景适配

#### 用户画像 C：情境驱动型用户
- 今天有特定需求，如一个人吃、和朋友聚餐、吃夜宵、想吃点热的、赶时间等
- 需要通过少量筛选快速得到合适结果

## 5. 核心痛点

### 5.1 信息繁杂，决策困难
- 周边餐饮选择很多，但缺少围绕“当前就餐情境”的快速决策工具。
- 传统平台以列表搜索为主，无法高效解决“我现在到底吃什么”的问题。

### 5.2 缺少针对大学生消费能力的产品设计
- 现有平台较少围绕学生预算、步行范围、宿舍生活与校园节奏做优先级排序。
- 用户实际更需要“低预算、高性价比、近距离、适合当前场景”的内容组织方式。

### 5.3 熟悉环境后的选择疲劳
- 学生高频在固定区域用餐，随着时间推移更容易产生“吃腻了”的心理。
- 即使有信息，也缺少能帮助用户快速随机决策的机制。

## 6. 解决方案概述

本产品通过两大核心功能来覆盖用户的两种主流决策场景：

### 6.1 功能一：今日食运
解决“**我懒得想，你替我做决定**”的问题。

核心机制：
- 用户先设置少量条件（如预算、步行时间、正餐/小食等）
- 系统基于条件生成一个候选池
- 用户可轻量编辑候选内容
- 进入电子摇签 / 抽签动画
- 输出今日运势与推荐餐厅
- 用户可选择“重新抽签”或“一键导航”

### 6.2 功能二：许愿
解决“**我大概知道想吃什么，但不知道去哪家**”的问题。

核心机制：
- 用户通过筛选项表达需求（预算、人数、区域、菜系等）
- 系统筛选并推荐 3-5 家符合条件的餐厅
- 页面以上方地图 + 下方推荐卡片的形式展示结果
- 用户可进入详情页或直接导航

## 7. MVP 范围定义

### 7.1 首版必须包含
- 大学城 / 学校选择
- 地图主界面
- 今日食运功能
- 许愿推荐功能
- 商家卡片与商家详情页

- 轻量搜索入口

- 完整账号体系
- 复杂个人主页
- 用户评论体系
- 实时商户抓取
- 复杂社交功能
- 完整 AI Agent 自主找店能力
- 交易闭环（下单 / 团购 / 优惠券）

### 7.3 数据范围建议
为保证 prototype 可跑通，首版建议：

- 仅选择 **1 个大学城**
- 仅覆盖 **40-80 家餐厅**
- 使用整理好的静态结构化数据
- 先验证信息架构和交互体验，再考虑扩区

## 9. 核心用户场景

### 场景 1：一个人临时决定吃什么
用户下课后不知道吃什么，希望在 1 分钟内得到建议。进入首页后点击“今日食运”，设置“步行 10 分钟内”“30 元以内”“正餐”，直接开签，获得一家推荐餐厅并导航前往。

### 场景 2：和朋友一起吃饭
用户与 2-3 位朋友在宿舍准备出门吃饭，希望找到适合多人、预算适中、适合坐下来聊天的餐厅。进入“许愿”，设置人数、预算、菜系和步行范围后，获得 3-5 家推荐结果并进行比较。

### 场景 3：有想法但没目标餐厅
用户今天想吃“热一点的、不要太贵、离学校近”，但没有具体餐厅目标。通过“许愿”完成结构化筛选，再查看推荐卡片。

### 场景 4：吃腻了周边餐厅，想用随机机制降低决策疲劳
用户平时经常在固定区域吃饭，希望“有控制地随机”。通过“今日食运”先限制预算和区域，再对候选池做少量增删，完成抽签决策。

## 10. 页面设计说明

## 10.1 首页

### 页面目标
作为主要入口页，承载产品核心心智：地图浏览 + 快速决策。

### 页面结构

#### 顶部常驻栏
- 左侧：大学城 / 学校切换入口
- 中间：**今日食运**主按钮（签筒图标 + 品牌名）
- 右侧：搜索按钮、头像按钮

#### 中部
- 地图主视图
- 默认展示当前位置 / 学校附近区域
- 地图上展示餐厅标记点

#### 底部面板
- 可切换或滑动展开
- 承载三类内容：
  - 附近浏览
  - 今日食运结果
  - 许愿推荐结果

### 设计原则
- 地图负责建立“附近有什么”的空间感
- 底部结果负责“帮我决定去哪家”
- 首屏必须突出“今日食运”这一品牌核心功能

---

## 10.2 今日食运模块

### 页面目标
帮助用户在有限约束条件下随机决策，降低选择困难。

### 功能定位
“有选择的随机”而不是“完全不可控的随机”。

### 交互流程
1. 用户点击首页顶部的“今日食运”入口
2. 进入快速筛选弹层 / 页面
3. 用户设置基础条件
4. 系统生成候选签筒内容
5. 用户可编辑签筒内容
6. 点击开始摇签
7. 播放电子摇签动画
8. 输出今日运势 + 抽签结果
9. 提供重新抽签 / 一键导航 / 查看详情等操作

### 首版建议筛选条件
- 步行范围：5 分钟 / 10 分钟 / 15 分钟
- 预算：20 元以内 / 30 元以内 / 50 元以内 / 80 元以内
- 类型：正餐 / 小食 / 甜品 / 夜宵
- 人数：1 人 / 2 人 / 3-4 人

### 建议增加的排除条件
- 今天不想吃辣
- 不想吃面
- 不要连锁
- 不想吃已经吃过的类型

### 候选签筒编辑
用户可在系统生成的候选项基础上进行轻量编辑：
- 删除某个不想吃的候选
- 手动增加一个菜品 / 餐厅候选
- 重新生成候选池

### 结果展示内容
- 今日运势文案（如：大吉 / 小吉 / 平签）
- 今日推荐餐厅名称
- 简短推荐理由
- 基础信息：人均、距离、步行时间、菜系
- 操作按钮：
  - 重新抽签
  - 查看详情
  - 一键导航

### 结果页设计目标
- 让结果看起来“好接受”
- 给用户情绪价值和轻娱乐感
- 同时保留明确的行动按钮，快速闭环

---

## 10.3 许愿模块

### 页面目标
帮助用户在已知偏好前提下，快速得到几家更适合的餐厅推荐。

### 功能定位
偏理性决策场景，不强调随机，强调条件匹配与结果比较。

### 输入方式
采用“结构化筛选 + 一句补充说明”的轻量模式，而非纯自然语言输入。

### 首版筛选项建议
- 用餐人数：1 人 / 2 人 / 3-4 人 / 5 人以上
- 人均预算：20 元以内 / 30 元以内 / 50 元以内 / 80 元以内
- 区域范围：步行 5 分钟 / 10 分钟 / 15 分钟
- 口味菜系：中餐 / 日料 / 韩餐 / 泰餐 / 西式 / 烧烤 / 麻辣烫 / 奶茶甜品等
- 场景类型：正餐 / 下午茶 / 夜宵 / 聚餐 / 赶时间快餐
- 补充说明：一句短文本，如“想吃热的”“想坐久一点”“不要太吵”

### 输出结果形式
- 页面上半部分：地图高亮推荐商家位置
- 页面下半部分：3-5 家推荐卡片

### 推荐卡片内容
- 店名
- 封面图
- 菜系
- 人均
- 步行时间
- 学生标签
- 推荐理由
- 操作按钮：查看详情 / 去这里

### 推荐逻辑原则
首版推荐逻辑应以**规则筛选 + 标签匹配**为主，结果稳定可控；如需 AI 能力，优先用于生成推荐理由文案，而不是完全依赖 AI 搜索结果。

### 页面内容

- 封面图 / 图片轮播
- 人均价格
- 距离 / 步行时间
- 菜系
- 营业时段
- 推荐菜
- 学生标签
- 适合场景说明
- 推荐理由
- 地图位置
- 操作按钮：
  - 一键导航
  - 加入签筒
  - 收藏（可选）

### 加入签筒功能说明
用户在浏览商家详情时，可主动将某家餐厅加入“今日食运”候选池，形成闭环。

## 10.5 搜索功能

### 页面目标
满足少量明确搜索需求，但不喧宾夺主。

### 搜索对象
- 餐厅名称
- 菜系 / 品类
- 标签关键词

### 搜索结果展示
- 地图高亮对应商家
- 下方展示匹配商家卡片列表

### 产品策略
搜索是辅助功能，不作为首页主路径，避免用户重新陷入传统平台式的“浏览过多信息”。

## 10.6 个人页 / 头像入口

### 页面目标
提供轻量个性化设置，不做复杂账号系统。

### 首版建议功能
- 当前学校 / 大学城信息
- 我的偏好设置（可选）
- 不吃什么 / 忌口设置（可选）
- 最近抽过的餐厅（可选）
- 收藏商家（可选）

### 产品策略
个人页首版只做轻量承载，不作为主要使用路径。

## 11. Onboarding 设计

### 页面目标
在不增加注册门槛的情况下获取推荐所需上下文。

### 首次进入建议流程
1. 选择所在大学城
2. 选择所在学校
3. 可选：是否授权定位
4. 进入首页

### 策略说明
- 首版不强制完整注册
- 学校信息可本地保存，作为推荐范围的重要参数
- 降低首次使用成本，提高主流程转化

## 12. 学生视角标签体系

这是产品差异化的关键内容，建议作为统一标签系统使用在商家卡片、详情页和推荐逻辑中。

### 建议标签
- 20 元吃饱
- 宿舍党友好
- 一个人也能吃
- 约饭不踩雷
- 适合赶时间
- 适合坐着聊天
- 夜宵救命
- 重口解压
- 清淡养胃
- 考试周续命
- 打卡型餐厅
- 性价比之选
- 聚餐友好
- 少排队
- 近食堂替代

### 使用方式
- 卡片上显示 1-3 个核心标签
- 详情页展示完整标签
- 食运与许愿的推荐逻辑可参考这些标签进行加权匹配

## 13. 推荐与筛选逻辑（MVP）

## 13.1 今日食运逻辑
1. 根据用户设置的预算、步行范围、类型、人数条件过滤候选餐厅
2. 根据排除项过滤掉不符合条件的结果
3. 从剩余候选中选取若干个生成签筒池
4. 用户可手动增删候选池
5. 从候选池中随机输出一个结果

### 优化建议
- 默认避免连续推荐最近已选中过的餐厅
- 默认避免连续输出完全相同品类
- 保证结果“随机但不过于离谱”

## 13.2 许愿逻辑
1. 根据结构化筛选条件过滤候选餐厅
2. 按标签匹配度、预算契合度、步行距离等进行排序
3. 输出 3-5 家推荐结果
4. 为每家餐厅生成简短理由说明

### AI 使用原则
- **首版**：规则负责出结果，AI 负责润色推荐理由
- **后续**：再考虑引入更复杂的 Agent 化推荐流程

## 14. 地图能力设计建议

### 地图在产品中的角色
- 提供“就在附近”的空间感
- 强化校园周边场景的本地属性
- 作为结果展示和导航前的确认层

### 首版地图能力建议
- 地图展示餐厅点位
- 支持高亮推荐商家
- 支持点击 Marker 查看基础信息
- 支持用户当前位置 / 学校位置展示
- 支持一键调起导航

### 技术策略建议
首版前端原型建议以 **高德 JavaScript 地图能力** 为主，优先实现可视化地图、点位展示与导航跳转，不必在首版深入复杂地图能力。

## 15. 数据结构建议

每家餐厅建议至少维护以下字段，以支持筛选、推荐与卡片展示：

| 字段名 | 说明 |
|---|---|
| id | 餐厅唯一标识 |
| name | 餐厅名称 |
| campus_area | 所属大学城 |
| school_scope | 覆盖学校范围 |
| latitude | 纬度 |
| longitude | 经度 |
| cuisine | 菜系 |
| category | 类型（正餐/小食/甜品/夜宵） |
| avg_price | 人均价格 |

## 17. MVP 成功指标

### 17.1 产品体验指标
- 用户进入首页后，能快速理解“这是一个帮我决定吃什么的工具”
- 用户能在较少点击下完成从筛选到出结果的闭环
- 推荐结果对学生用户具备明显的“场景适配感”

### 17.2 行为指标（可用于演示或后续埋点）
- 首页到食运 / 许愿的点击率
- 今日食运完成率
- 今日食运结果接受率
- 许愿推荐结果点击率
- 商家详情页进入率
- 一键导航点击率

### 17.3 MVP 验证重点
重点不是验证“信息是否全”，而是验证：
- 用户是否更快做出决策
- 产品是否真的更像一个“懂学生”的吃饭工具
- 食运机制是否足够有趣且不失控

### 19.1 个性化能力

## 20. 最终产品结论

「今日食运」的首版产品目标，不是做一个“大而全”的美食信息平台，而是做一个真正能帮助大学生快速决定吃什么的校园决策工具。

产品的核心竞争力来自三点：

1. **地图化的本地空间感**：告诉用户附近有什么。
2. **今日食运的轻娱乐决策机制**：替用户降低决策成本。
3. **学生视角的标签与推荐逻辑**：让推荐真正贴近校园生活。

只要首版能够让用户以低成本完成“打开页面 → 得到建议 → 去吃饭”的闭环，这个 prototype 就已经具备了明确的产品价值和展示说服力。

## 21. 可直接交付 AI Coding 的简版需求摘要

### 产品名称
今日食运

### 产品类型
大学城美食地图网页应用（前端展示型 Prototype）

### 首页结构
- 顶部栏：学校切换 / 今日食运 / 搜索 / 头像
- 中间：地图
- 底部：结果面板

### 核心功能 1：今日食运
- 用户先选预算、步行时间、类型、人数
- 系统生成候选池
- 用户可删除或增加候选项
- 点击开始摇签
- 输出运势文案 + 推荐餐厅
- 提供重新抽签 / 查看详情 / 一键导航

### 核心功能 2：许愿
- 用户选择人数、预算、范围、菜系、场景
- 系统推荐 3-5 家餐厅
- 上方地图高亮位置，下方展示推荐卡片

### 详情页
- 店名、图片、人均、距离、标签、推荐菜、推荐理由、导航按钮、加入签筒

### 数据策略
- 首版只做 1 个大学城，40-80 家餐厅
- 使用静态 mock 数据
- 先规则筛选，后续再叠加 AI 推荐理由
